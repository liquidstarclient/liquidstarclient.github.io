jsua / quick-control.js
jsua.register({ name: "Quick Control" }, function init() {
  // one control, one action, one cleanup
  kit.menuFeature("Visuals", "Weapon Chams");
  jsua.menu.button("Look at enemy", function () {
    const enemy = players.closestEnemy();
    if (enemy) self.lookAt(enemy);
  });
}, function cleanup() {
  kit.off(["Weapon Chams"]);
});

Start · Introduction

Welcome to JSUA

JSUA is the part of Liquid Star where you add your own ideas to the client. This page is the gentle start — what it is, where scripts go, and how to get one running. No coding knowledge needed here; the deeper wedges handle that.

§1 What JSUA Is

JSUA stands for JavaScript User Application. In plain terms it is a small script — a short list of instructions — that the client runs for you. People use scripts to add little quality-of-life tweaks, to automate a small action, or to tie a few existing features to a single button.

You do not need to be a programmer to use one. You only need the script file itself, and a place to put it.

§2 Where Scripts Live

Every JSUA is a plain text file that ends in .js. They sit in a folder named jsua, created next to the client. The loop is short: drop a script file into that folder, open the JSUA tab inside Liquid Star, and your script appears in the list. Tick it on to enable it, tick it off to disable it.

§3 You Don't Have to Code

This is the important part. If you can't write code, you don't have to. The fastest path is to let an AI write the script for you. Hand it the AI Reference file linked at the top of this page, describe what you want in normal words — like "warn me when my ammo is low" — and paste the result back into a file. The reference tells the AI exactly what it is allowed to use, so the script stays valid.

§4 How to Read the Rest

The dial above is your table of contents. Each wedge opens one short article. This Start wedge is just the welcome; the others go deeper — Anatomy of a Script shows how to build one and add buttons, Reading the Match lists every command you can use, Interacting with the Client shows how to flip features and test as you go, The Sandbox explains the safety rules, and Deploying Bots connects extra players you control. Read them in order, or jump straight to the one you need.

Scripts · Foundations

Anatomy of a Script

Different scripts do wildly different things, yet under the hood they all share the same simple shape. Learn that shape once and the rest is just filling in the blanks.

§1 The Register Wrapper

A script comes to life the moment you call jsua.register(meta, init, cleanup). It always takes the same three things in the same order. meta is a small label — at the very least a name. init is a function that sets things up when the script is switched on. cleanup is a function that puts everything back when it is switched off.

The client runs init once when you tick the script on, and cleanup once when you tick it off. Between those moments nothing of yours runs unless you asked for it — so a tidy script leaves no trace once it is off.

§2 Adding Controls — Checkboxes, Sliders, Buttons

This is where most scripts get useful. Inside init you can build a small panel that appears under your script in the JSUA tab. There are four building blocks, and you create them right away (not inside a timer):

Every control returns a small handle with getValue() and setValue(), so you can read or change it later. And you never tidy them up by hand — all menu controls vanish automatically when the script is disabled.

§3 Talking to the Player

Two helpers let a script speak. jsua.notify(message, type) shows a small toast, where type is "success", "info", "warning", or "error". jsua.log(name, value) writes a line into the JSUA Console — the place you check what a value really is while you are testing.

§4 Repeating Actions Safely

Most scripts repeat something. Rather than reaching for a raw setInterval, use kit.every(ms, fn) to repeat an action and kit.after(ms, fn) to run one once after a delay. The reason matters: these stop on their own when the script is disabled. Timers, listeners, and menu controls are all cleaned up for you — the only thing you undo by hand is a client feature you switched on.

State · Reference

Reading the Match

This is your lookup table. Every command JSUA gives you to read the game or act in it is listed below, grouped by what it touches, with a plain note on what each one does. Skim it once and you will know what is possible.

§1 You — the self object

self is your own player. Some of these read a value; the bottom three do something.

self.posYour position right now, as {x, y, z}.
self.yawWhich way you face left/right, in radians.
self.pitchWhich way you face up/down, in radians.
self.healthYour current health.
self.teamYour team number (used to tell friend from foe).
self.press(key)Hold a key down — name like "d", "space", "ctrl", or a code like 68.
self.unpress(key)Release a key you pressed. Always pair it with a press.
self.lookAt(target)Turn your view toward a player, a point, or an attached object.
self.serverRotationThe angle the server currently sees you facing, as {yaw, pitch}.
self.setServerRotation(yaw, pitch)Make the server see a fixed angle while your screen keeps your real aim (silent-aim-style). null stops it.
self.compMovement(on)While a server rotation is set, keep walking where you actually look instead of toward the spoofed angle.
self.fakeCamera({yaw, pitch})Visual only: point your on-screen view somewhere while the server keeps the real angle. null stops it.

§2 Everyone else — the players object

Always the live list of other players. The filters save you from sorting by hand; the closest… calls hand back a single player.

players.allEveryone in the match except you.
players.enemyOnly players on other teams.
players.friendlyOnly your teammates.
players.closest()The nearest player to you of any team.
players.closestEnemy()The nearest enemy. The one you usually want.
players.closestFriendly()The nearest teammate.

§3 Your weapon & aim — weapon and calc

JSUA gives the game's renamed fields steady names here, plus a few numbers it works out for you.

weapon.ammoRounds left in the current weapon.
weapon.reloadingtrue while you are reloading.
weapon.shotDelayThe gap between shots for this weapon.
calc.lookDirThe direction you are looking, as a unit {x, y, z}.
calc.lookYaw / calc.lookPitchYour current aim angles, ready to use.

§4 Helpers — kit

Small tools that make the rest easier. kit.ready() is the one you will use most — it tells you the match actually exists yet.

kit.ready()true once you have spawned and the player exists. Guard match logic with it.
kit.pos(target)Pulls a clean {x, y, z} out of a player or point.
kit.distance(a, b)How far apart two players or points are.
kit.every(ms, fn)Run fn over and over; stops itself on disable.
kit.after(ms, fn)Run fn once after a delay.
kit.key(key, down, up)Bind a hotkey; it unbinds itself on cleanup.
kit.clamp(v, min, max)Keep a number inside a range.
kit.lerp(a, b, t)Blend smoothly between two numbers.

§5 Putting it together

Reading and acting in one small loop. Notice the kit.ready() guard — before you spawn, the values above are empty, so this keeps the script quiet until the match starts.

Interact · Control

Interacting with the Client

Interacting means flipping the client's own switches from a script — the same toggles and sliders you see in the menu. The trick to learning it is not memorising names; it is printing what's there once, then writing from what you saw.

§1 What You Can Interact With

Every feature in the client carries a text label — exactly the words you read in the menu, like "Weapon Chams" or "Third Person". The feat object is your remote control for them. There are only three verbs to learn: read one, set one, or list them all.

feat.list()Show every feature you can touch right now, with its label and value.
feat.get(label)Read what one feature is currently set to.
feat.set(label, value)Turn a feature on with true / off with false (or set a number).

One rule is built in: feat always obeys your account tier. Features above your tier never appear, so a script can never reach past what your account is allowed.

§2 Pull It Once, Then You Know

You should never guess a label. Instead, run one debug print and read the answer — after that, you know every name the rest of your script needs. Drop a temporary button into your panel, enable the script, click it, and look at the JSUA Console.

The console now lists the real labels. Copy the one you want, and you are ready to control it for real.

§3 Now Make It Do Something

With the label in hand, controlling it is one line. Here a hotkey toggles Weapon Chams on and off, and the script cleans up after itself — the golden rule from The Sandbox: turn off what you turned on.

§4 Shortcuts for Common Cases

When you do the same thing a lot, kit has a one-liner for it — convenience only, never extra power.

kit.menuFeature(label, feature)Make a checkbox already wired to a client feature.
kit.loadout({ ... })Turn a whole set of features on/off in one call.
kit.off([labels])Switch a list of features back off together — perfect for cleanup.

Safety · The Boundary

The Sandbox

"Sandbox" is just a name for a safe, walled-off play area — like a child's sandbox, you can build anything you like inside it, but nothing you do leaks out into the rest of the computer. This article explains that boundary and why it is the reason you can run scripts at all.

§1 What "Sandbox" Means

Your script runs inside a sealed area. From inside it you can see and change the game freely — players, features, your aim, the screen. But you cannot reach outside the game to your files, the wider internet, or other programs. That invisible wall is the sandbox. The name comes from exactly that picture: a box you play in safely.

§2 The Role It Plays

The sandbox is not there to annoy you — it is what makes JSUA safe to use at all. It does three jobs: it stops a downloaded script from harming your computer, it lets you trust scripts other people share, and it enforces your account tier so a script can never unlock features you do not own. Without it, running a stranger's script would be reckless. With it, the worst a bad script can do is not work.

§3 Inside the Box — What You Can Do

There is plenty of room to build: ordinary JavaScript, timers and listeners, on-screen overlays you draw yourself, reading files from the jsua folder, everything in the other articles, and WebSocket connections to voxiom.io only. For nearly anything you will want to make, the box is big enough.

§4 The Walls — What Is Refused

Before a script runs, the client scans it. If it finds a way to escape the box it refuses to load the script at all — nothing half-runs. The blocked moves are require, fetch, XMLHttpRequest, eval, new Function, web workers, and Electron bridges. If an AI ever hands you code with one of these, just ask it to redo the task using only the JSUA API.

§5 The Tools Are Already Inside

So you do not reach out for libraries, JSUA hands you everything ready to use: jsua, game, self, players, weapon, feat, calc, kit, bot, on, and off. Use them freely — just never redeclare one with let or const, or you will hide the real thing. And one promise to keep in return: turn off every feature you turned on, and release every key you pressed, inside cleanup.

Bots · Deploy

Deploying Bots

The bot object connects extra players to your lobby as separate guests — not your account — and lets your script drive them: move, aim, shoot, switch weapons, respawn. Each bot is its own connection, so you can run several at once.

§1 Connect and Keep the Handle

bot.connect() joins the lobby you are in and hands back a controller. Keep it in a variable and call everything on that variable — there are no ids to look up. Connect again for a second bot, a third, and so on. Bots are shared and keep running until stopped, so always end with bot.disconnectAll() in cleanup.

const b = bot.connect()Join the current lobby; returns this bot's controller.
bot.list()Every live controller, as an array.
bot.countHow many bots are connected.
bot.disconnectAll()Drop every bot. Put this in cleanup.

§2 Driving One

Everything below is called on the handle you kept (here, b). Movement keys and an action can happen in the same moment — a bot can walk and shoot at once.

b.press(key) / b.unpress(key)Hold/release "w" "a" "s" "d" "space" "shoot" "aim".
b.tap(key, ms)Press, then release after ms (default 120).
b.yaw / b.pitchSet view angles in radians (pitch up is positive, clamped to ±1.5).
b.lookAt(target, from?)Aim at a player or point. Defaults to aiming from your position.
b.slot(n)Switch to weapon slot 0–4.
b.shoot(on) / b.shootOnce()Hold/release fire, or fire a single shot.
b.jump()Jump once.
b.respawn()Respawn the bot.
b.disconnect()Drop just this bot.

§3 Three Bots at Once

One shoots the nearest enemy, one stares up and spins, one strafes back and forth. Notice each bot is just a variable you drive.

Want an AI to write it? Hand it the full reference and ask for one small behavior.
